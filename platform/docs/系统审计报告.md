### [P0-A1] Job 哑死检测指标因 UTC/本地混用永久失真（可掩盖资金 Job 停摆）
- **位置**：写入 `jobs/index.js:14`（`last_run_at = datetime('now','localtime')`，本地）与 `jobs/index.js:25`（`last_success_at = new Date().toISOString().replace('T',' ').slice(0,19)`，**UTC 墙钟去 Z**）；读取 `routes/metrics.js:21`（`ageSeconds = (Date.now() - new Date(ts.replace(' ','T')).getTime())/1000`，无 Z 串按**本地**解析）、`metrics.js:55`（`job_last_success_age_seconds`）。
- **现象**：同一张 `job_runs` 表，`last_run_at` 存本地、`last_success_at` 存 UTC；`/metrics` 用同一个 `ageSeconds` 解析两类列，但只有 `last_success_at` 是 UTC。
- **证据**：东八区某刻真实 UTC=12:00、本地=20:00。`last_success_at` 写成 "12:00"；metrics 把 "12:00" 当本地（=04:00 UTC）解析 → `age = 真实now(12:00 UTC) − 04:00 UTC = 8h`。即**每个 Job 的 age 恒被抬高约 28800 秒**。
- **影响**：哑死规则"`age > 2×周期`即告警"对 5 分钟级 Job（结算重试、提现、回调补单）意味着 age 永远 ≥8h ≫ 阈值 → 指标**长期红着**，真停摆淹没在基线噪声里 → 资金类 Job 哑死实质失去监控。仅当进程 TZ=UTC（容器常见）时偶合无偏差，掩盖了问题的存在。
- **复现/推理**：部署 `TZ=Asia/Shanghai`，启动任一 Job，抓 `/metrics` 看 `job_last_success_age_seconds{job=...}` 即恒 ≈28800+。
- **修复建议**：`jobs/index.js:25` 的 `last_success_at` 改为与 `last_run_at` 一致的 `datetime('now','localtime')`（最小改动）；根治见 [B4] 全栈 UTC。
- **是否触动 npm test 契约**：否（无测试断言 metrics age）。需重跑确认。

### [P1-A2] PII 导出审批闸门可被自报行数绕过 + 下载无视申报范围导出全量名册
- **位置**：`adminOps.js:325-350`（申请+自动批准，`:337 autoApprove = rowEstimate != null && rowEstimate < threshold`）、`adminOps.js:393-419`（下载，`:404-408` `SELECT ... FROM users u ... WHERE u.role='worker'` 忽略 `e.scope/e.row_estimate`）。
- **现象**：① 自动批准依据**申请人自报**的 `rowEstimate`；② 下载查询与申报的 `scope/rowEstimate` 完全无关，恒导出**全部**零工的**完整手机号**+主体/锁定/信用/本年收入。
- **证据**：`autoApprove` 为真即把 `status='approved'`；`/exports/:id/download` 仅校验 `applicant_id===req.user.id` 与未过期，随后 dump 全表。持 `worker:read` 的角色有：财务税务/风控专员/只读审计/客服（`db.js:931-934`）。
- **影响**：任一上述角色账号声明 `scope="测试", rowEstimate=1, reason=...` → 命中自动批准 → 立即下载**全量零工完整 PII**，48h 内可重复下载。**双人审批（SoD）形同虚设**，违反 PIPL 最小必要/审批原则。水印 `EXPORT-{id}` 与审计日志仅为事后取证。
- **关系**：目标把"②导出 rowEstimate<阈值自动批准且可重复下载"列为既定意图（A 轴 current behavior）。**但"下载忽略 scope/rowEstimate 一律全量"超出了'小批量自助'的本意**——闸门不是放宽，而是被架空。故定 A 轴 P1。
- **复现**：`POST /admin/exports {scope:'x',reason:'xxxxx',rowEstimate:1}` → `GET /admin/exports/{id}/download` 返回全量。
- **修复建议**：(a) 下载必须按 `scope`（如指定企业/时间范围）裁剪结果集，且行数与申报 `rowEstimate` 一致性校验（实导 > 申报×系数即拒绝并告警）；(b) 自动批准阈值不应信任自报，应对"将实际命中行数"在批准时由服务端预估；(c) 全量名册类导出一律强制审批（见 [B8]）。
- **是否触动 npm test 契约**：**是**。`audit_fixes.test.mjs`（约 L104/112）固化了"rowEstimate<阈值自动批准+可复下"。收紧需同步改测试与 API.md，影响面已标注。

### [P1-A3] 阈值锁两站点对个体户判定不一致 → 个体户被反锁且无自助解锁
- **位置**：`risk.js:95-110`（`postSettlementChecks` 滚动12月累计达阈值即 `UPDATE worker_profiles SET locked=1, lock_reason='threshold'`，**无 subject_type 守卫**），被 `settlement.js:265` 对**每笔**结算无条件调用；对照 `jobs/housekeeping.js:47-52`（每日兜底，`WHERE p.subject_type='person' AND p.locked=0`，**正确排除个体户**）；自助解锁 `worker.js:254-275`（`:259 if subject_type==='soletrader' throw ALREADY_SOLETRADER`）。
- **现象**：B 线（个体户）零工每笔结算也会进 `postSettlementChecks`；其 `tax_records.gross`（含经营所得）计入滚动12月累计，达 ¥480万即被打 `threshold` 锁。但个体户调 `/worker/soletrader` 自助解锁会被 `ALREADY_SOLETRADER` 拒绝 → **无路可解**。
- **证据**：`postSettlementChecks` 滚动查询 `SELECT SUM(gross) FROM tax_records WHERE worker_id=? AND period>=?` 无 `method/subject` 过滤；`housekeeping` 同规则却带 `subject_type='person'`。两站点对"是否锁个体户"结论相反。
- **影响**：① **走完"强制登记→注册个体户→解锁"正路的零工，在其后第一笔 B 线结算即被重新锁定**，且无法自助解除，只能提工单等运营手动解锁——而 `threshold` 锁语义上被设计为"零工自助解除"，运营可能不主动处理 → 卡死接单。② 个体户还会收到无意义的"集中度/个体户引导"预警（`risk.js:82-93`）。阈值 ¥480万虽高，但**正中目标用户（高收入转 B 线者）**，可达性高。
- **复现**：让一个 person 累计达阈值被锁 → 注册个体户解锁 → 再完成一笔 B 线结算 → `worker_profiles.locked` 再次=1，`/worker/soletrader` 返回 ALREADY_SOLETRADER。
- **修复建议**：`postSettlementChecks` 强制阈值锁分支加 `subject_type='person'` 守卫（与 housekeeping 对齐）；并把集中度/引导检查也限定自然人。
- **是否触动 npm test 契约**：需重跑确认。`audit_fixes2` D3 测的是 person 被锁/个体户登记自助解锁，加 person-only 守卫不影响该用例；但建议补一条"个体户高收入不被 threshold 锁"的回归。

### [P1-A4] T+1 对账"逐笔核销"对结算腿存在粒度错配 → 不平日产生大量幻影差异
- **位置**：`jobs/dailyRecon.js:16-19`（平台侧 `fund_flows.type IN ('recharge','settle_out','withdraw')`）、`:46-60`（按金额多重集合相消生成逐笔差异）；对照结算腿 `settlement.js:159-187`（每笔结算调用 `escrow.transfer` **3 次**：net/tax/margin → 3 条 `escrow_txns`）。
- **现象**：一笔结算在**银行侧产生 3 条流水**（net、tax、margin），但**平台侧只 1 条** `settle_out`（金额=charged=gross+margin）。
- **证据**：总额对齐（net+tax+margin = charged，`status='balanced'` 头部判断正确）；但逐笔差异仅在 `status='mismatch'` 时跑，且以**金额**相消：银行 `[net,tax,margin]` 与平台 `[charged]` 金额各不相等，互不相消 → net/tax/margin 全落 `bank_only`、charged 落 `platform_only`。
- **影响**：充值（1:1）、提现（1:1）正常；**唯独结算 3:1**。一旦某日总额真不平（真问题日），当日**每笔结算**都会被误判为多条逐笔差异，"对账差异工作台"被幻影项淹没——恰在最需要它的时候失真。另外逐笔仅按金额相消、不关联 ref，偶合同额会错误抵消不相关流水。
- **复现**：构造某日 1 笔结算 + 1 处真实掉单使总额不平，跑 `runDailyRecon` → `recon_diffs` 出现 net/tax/margin/charged 等幻影行。
- **修复建议**：逐笔核销应在**可比口径**上做——要么平台侧按腿展开（结算落 3 条对应流水或在 recon 时拆 charged=net+tax+margin），要么银行侧按 `idem_key/confirm_no` 聚合回结算单维度；相消应带 `ref_no/confirm_no` 关联而非纯金额。
- **是否触动 npm test 契约**：可能。`commercial.test.mjs` 覆盖对账，需重跑；若仅改 mismatch 分支逐笔逻辑、保持总额口径，主用例大概率不受影响。

## 3. B 轴 — 设计与架构优化建议（逐条）

### [B1] 资金账务：应采用不可变复式记账（double-entry ledger）+ 资金接口幂等键
- **现状**：`accounts(balance,frozen)` 可变余额 + `fund_flows` 每账户**单边**流水。守恒由代码保证（`accounts.js#apply` 三重守卫 `balance/frozen/balance-frozen ≥0`；`settlement.js` 用 `net=gross-tax-vat` 残差法 + `charged=gross+margin`）。并发安全靠"重读最新态+条件更新"（better-sqlite3 同步执行，单进程下成立）。
- **为何不是最优**：生产级资金/税务系统的最佳实践是**不可变复式账本**——每笔资金事件是一组借贷平衡的分录、带统一 transaction_id，账户余额由分录推导而非可变字段。当前模型：① 守恒是"约定"而非"结构性保证"（任何新写资金的代码都可能破坏）；② `fund_flows` 单边记账难以一眼校验每笔事件全局平衡；③ 余额可变字段缺乏"由流水重放可重建"的强校验。`/metrics` 的 `balance<0 OR frozen<0 OR balance<frozen` 是事后探针，不是事前结构约束。
- **改进方案**：引入 `ledger_entries(txn_id, account, debit, credit, ...)` 不可变分录表，单笔业务=一组 `SUM(debit)=SUM(credit)` 的分录；余额=分录聚合（或物化视图 + 定期重放校验）。资金类 API（accept/withdraw/recharge）增 `Idempotency-Key` 头，幂等键落库去重（现仅 `escrow.transfer` 层有 idemKey，入口层无）。
- **影响面**：schema 新增账本表 + 迁移既有 `fund_flows`；`accounts.js`/`settlement.js`/`escrowEvents.js` 改写为记分录；测试需大改（integration/commercial/audit_fixes2 大量断言账户余额）。**数据迁移：需要**。
- **收益/风险**：收益=可审计性、对账可靠性、守恒结构化；风险=改动面大、契约重写。**优先级 P1（战略级，可分阶段：先加分录表与入口幂等键，再逐步以分录为真相源）。**

### [B2] 数据存储：SQLite(WAL) 不适配"多实例资金系统"，应评估迁移 PostgreSQL
- **现状**：`db.js:9-11` 仅 `journal_mode=WAL + foreign_keys=ON`，**未设 `busy_timeout`**；`config.js:44` 暴露 `ROLE=api`（多实例水平扩展）；README 称"API 实例 ×N + Worker ×1"。
- **为何不是最优**：① SQLite 是**本地文件**，多台 API 实例无法共享同一库文件（网络文件系统会破坏其锁语义）——文档的水平扩展与 SQLite 根本冲突；② 即便同机多进程，缺 `busy_timeout` 时并发写会直接 `SQLITE_BUSY` 抛错（"重读+条件更新"的乐观并发在跨进程下退化为偶发失败）；③ 生产资金系统对并发写、持久性、备份恢复（热备/PITR）、水平扩展的要求，SQLite 难以满足。
- **改进方案**：(a) 单机部署下立即加 `db.pragma('busy_timeout=5000')` 兜底；(b) 中期评估迁移 PostgreSQL（schema 基本兼容，README 已提"可平滑迁移"）——届时复式账本（[B1]）配合行级锁/`SELECT ... FOR UPDATE` 才能真正支撑高并发资金场景。
- **影响面**：busy_timeout 一行改动、零迁移；Postgres 迁移=连接层/SQL 方言/迁移工具引入，工作量大。
- **收益/风险**：收益=并发正确性与扩展性；风险=迁移成本。**优先级：busy_timeout=P1（低成本高收益）；Postgres 评估=P2（战略）。**

### [B3] API 设计：资金类操作应支持 `Idempotency-Key` 头；版本化与错误模型已较好
- **现状**：错误模型统一 `{error:{code,message}}`、分页统一 `?page&pageSize→{list,total}`（四端契约子审计确认一致性很高）；路径前缀 `/api/v1`。但资金写接口（accept/withdraw/recharge-orders）**无客户端幂等键**，依赖服务端状态机去重。
- **为何不是最优**：网络重试/重复提交是支付系统常态，行业标准（Stripe 等）以 `Idempotency-Key` 请求头在入口层去重。当前虽有状态机兜底，但入口层缺标准幂等契约。另：`updated_at` 蛇形泄漏（P3-A18）破坏 camelCase 一致性；`/api/v1` 之外无明确版本演进策略。
- **改进方案**：资金类 POST 接受 `Idempotency-Key`，落 `idempotency_keys` 表（键→首次响应）；统一序列化层强制 camelCase；明确弃用/版本策略。
- **影响面**：中间件 + 表；前端可渐进采用。**优先级 P2。**

### [B4] 时区：全栈统一 UTC，仅展示层转本地
- **现状**：约 6 处 `expires_at/deadline/last_success_at` 用 UTC 写（refresh_tokens/sms_codes/login_attempts.locked_until/export_requests/disputes.stage_deadline/job_runs.last_success_at），全库 50+ 处 `created_at` 等用 `datetime('now','localtime')`。逐处比较**多数碰巧两端对齐**（UTC/UTC 或 local/local），但 [P0-A1] 已是对齐被打破的实例；`disputes.stage_deadline`（UTC）若被"顺手统一成 localtime"会令协商/举证/公示期集体晚 8h 到期（零工资金多锁 8h）。
- **为何不是最优**：本地时间+UTC 混存是公认反模式；当前安全完全依赖两条隐式前提（每处比较两端被同一开发者用同一时区 + 进程 TZ==SQLite localtime TZ），无任何代码层强制。
- **改进方案**：所有写入 `datetime('now')`（UTC），所有 JS 比较走 UTC，展示层（web-admin `fmtTime`、web-company `format.js`、小程序）统一 UTC→本地转换。
- **影响面**：50+ 处默认值改写 + 既有数据 +8h 回填 + 前端展示层改造 + 测试时间构造对齐。**数据迁移：需要。** **优先级 P1（先修 P0-A1，再整体统一）。**

### [B5] 税务合规：税档硬编码、规则无生效日期版本化
- **现状**：七级累进税档**硬编码**于 `taxEngine.js:11-19`（`BRACKETS`）；费用减除率/月减除/增值税免征额/征收率在 `system_configs`（可运行时改，`db.js:950-996`），但**改了即全局生效、无版本与生效日期、无历史回溯**。
- **为何不是最优**：税收政策随时间变（起征点、税率、免征额历史均有调整）。最佳实践是**规则随生效日期版本化**，历史结算按当时规则可回溯重算；当前改 `vatFreeMonthlySales` 会影响对历史月的任何重算，且税档变更必须改代码发版。
- **改进方案**：税档与税务参数纳入带 `effective_from/effective_to` 的版本化规则表，算税按"所属期对应的规则版本"取数；保留历史版本以支撑汇算/稽查回溯。
- **影响面**：新增税规则版本表 + `taxEngine` 取参改造 + 迁移现有配置为 v1。**优先级 P1（合规刚需）。**

### [B6] 前端架构：web-admin 无中央 label 映射是可维护性硬伤
- **现状**：web-company 集中在 `src/utils/format.js`+`StatusTag.vue`；**web-admin 无中央映射**，29 个 view 各自内联 map（`format.js` 仅 fmtMoney/fmtTime）；小程序每页内联。字典子审计据此定位到 8 处漂移（P1-A5 语义反向、tasks.status 文案、争议/工单等）。
- **为何不是最优**：同一枚举在三端各写一套 → 必然漂移（已实证）；新增枚举值或改文案易只改一处。
- **改进方案**：① 后端新增 `GET /meta/dictionaries` 统一下发"枚举值→中文 label"，三端消费同一份；或 ② 至少 web-admin 建中央 `labels.js`（对齐 web-company 模式），各 view 引用。优先方案①（单一事实源，与 system_configs 字典分发理念一致）。
- **影响面**：后端一个端点 + 三端改造引用；纯前端无 server 测试风险。**优先级 P1（治本，消除整类漂移）。**

### [B7] 状态机：散落在 route/service，应抽出集中式定义
- **现状**：task/settlement/dispute/withdrawal/ticket 等状态转移**散落**在各 route/service 的条件更新里（如 `hireWorker` `WHERE status='recruiting'`、`disputes.js` 各转移、`settlement` 状态流）。守卫到位（已验证无非预期死路），但定义分散。
- **为何不是最优**：合法转移集合无单一声明源，新增转移/审阅可达性需通读多文件；易引入非法转移（历史上的 rejected/closed 死路即此类，已修）。
- **改进方案**：抽出集中式状态机声明（每实体 `{state: {event: nextState, guard}}`），转移统一经状态机校验，非法转移由结构拒绝。
- **影响面**：重构 service 转移点；测试需对齐。**优先级 P2（已较稳，属健壮性投资）。**

### [B8] 安全：PII 导出审批机制设计（与 A2 同源的 B 轴判断）
- **现状**：自报 `rowEstimate<阈值` 自动批准 + 下载无视 scope 全量导出（见 [P1-A2]）。
- **为何不是最优**：违反 PIPL"最小必要 + 审批留痕 + 范围受限"。审批应基于**服务端可验证的范围与规模**，而非申请人自报；导出结果必须被申报范围约束。
- **改进方案**：见 [P1-A2] 修复建议；并将"全量含完整手机号名册"列为**强制双人审批**类，禁止自动批准；下载列默认脱敏，完整手机号单列另需 `user:read_pii`。
- **影响面**：触动 audit_fixes 的②契约（同 A2）。**优先级 P1。**

### [B9] 迁移管理：db.js 自研幂等迁移，无版本与回滚
- **现状**：`db.js` 用 `CREATE IF NOT EXISTS` + `hasColumn` 守护的 `ALTER TABLE` + `rebuildIfCheckMissing`（12 步重建）做幂等迁移，单文件、导入即跑、无迁移工具。
- **为何不是最优**：无迁移版本号/历史/回滚；CHECK 约束变更靠整表重建（已正确处理外键，但脆弱）；多人协作易冲突。
- **改进方案**：引入轻量迁移框架（如 `node-pg-migrate`/`umzug` 或自建 `schema_migrations` 版本表 + 有序迁移文件）。配合 [B2] Postgres 一并落地更顺。
- **影响面**：迁移流程改造，不改业务。**优先级 P2。**

## 4. 经评估已足够好、暂不建议改动（防误改清单）

> 理由均为"**设计本身确实合理**"，非"测试如此固化"。

1. **资金守恒（残差法）**：`settlement.js:117 net=gross-tax-vat`、`:226 charged=gross+margin`、`finalizeSettlement` 中 `unfreeze(refund)+settleOut(charged)` 恰好释放 `price`、各账户 credit 之和=charged。逐项推演**严格守恒，无浮点漂移**（`money.js` 全整数分，`mulRate` 仅用于可容忍取整的 margin/vat 增量）。**勿改为各项独立 mulRate**，会破坏守恒。
2. **两阶段结算幂等**：`processSettlement:135` 重读最新单、`:137` done 直接返缓存、逐腿 `idemKey`、`finalizeSettlement:232` 事务内复查 done。并发双 finalize/重试不双花——`audit_fixes2` 已固化且设计正确。
3. **四流硬校验**：`acceptAndSettle:34-56`（争议锁/工单/交付物/状态/B线票）无可绕过路径。**勿加旁路。**
4. **争议锁与执行幂等**：`createDispute:81-92` 事务内条件抢 `dispute_id`；`executeDispute:185` 终态早返 + `:192 alreadyExecuted=!!closed_at` 守护 no_pay 无屏障解冻；pending 结算时保留锁待下轮收尾。设计严谨。
5. **结算重试有界恢复**：`settlementRetry.js:11` 重推 `pending+failed AND attempts<8`，与 `settlement.js:203` 3 次转 failed、`:207` 在 3/8 两点告警的设计自洽。
6. **累计预扣（16 号公告断月重置）**：`taxEngine.js consecutiveRun/calcWithholding` 断月即止、`tax=max(0,cumTax-segTax)`、period 取数与存储均本地时间**自洽**。增值税 10 万免征增量法正确。
7. **提现资金回收**：申请时 `freezeRef`（`worker.js:799`），成功 `withdrawFrozen`、失败 `unfreezeRef`（`escrowEvents.js:53-79`），终态幂等。守恒。

## 6.2 B 轴：已落地 / 战略暂存（含暂存理由的反思）
**已落地**：B2(busy_timeout=5000 多进程写锁兜底)；B6(关键 label 漂移修复 + 三端 RBAC 一致)；B4(P0 时区错配修复，关键路径一致 localtime)；多项 A 轴修复本即 B 轴最佳实践（最小权限、对账粒度、勾稽口径、窗口统一）。

**战略暂存**（附为何不在单次会话盲改的反思）：
- **B1 复式不可变账本 + 资金幂等键**：当前模型经审计**验证严格守恒**(§4)且测试全绿。全量改造触及每个资金操作 + 全部断言 + 数据迁移，属多阶段迁移。最优是分阶段：已做(守恒验证+负余额监控+busy_timeout)→ 增 `ledger_entries` 不可变表与入口幂等键 → 切真相源。不宜单会话盲改资金核心。
- **B2(Postgres)**：SQLite 本地文件与 `ROLE=api×N` 水平扩展根本冲突，迁移属部署级决策(连接层/方言/迁移工具/压测)，非代码内可验证；已加 busy_timeout 缓解单机多进程。
- **B4(全栈 UTC)**：**反思后维持**——对单时区(中国)产品，一致 localtime 即正确最优；全栈 UTC 收益面向多时区(本产品无此需求)，迁移需改 50+ 默认值 + 数据+8h 回填 + 前端展示层 + 测试时间构造，风险/收益不划算。已修唯一真错配(P0-A1)。
- **B5 税规则按生效日期版本化**：合规价值高但触及已验证正确的税引擎 + 边界测试。最优是建 `tax_rules(effective_from)` 表 + 按所属期解析版本 + 历史可回溯重算（这是有价值的独立特性）；把 BRACKETS 挪成带日期常量的"半措辞"改动并不能交付"历史重算"真正收益、反增间接层，故不做 token 式改动，建议作下一专项。
- **B3 API 幂等键 / B7 集中状态机 / B9 迁移框架 / B6 全量中央字典**：均为结构性增强/重构，现有守卫已证无双花(§4)，建议独立专项落地。

战略项优先级：**B5 税规则版本化(合规) > B1 复式账本(可审计) > B2 Postgres(扩展)**，每项可单独立项 + 配套测试。
